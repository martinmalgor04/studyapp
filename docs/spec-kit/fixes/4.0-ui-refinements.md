## 4.1.1 - Supabase Query JOINs

**Fecha**: 28 Enero 2026  
**Sprint**: 4.0 - UI Refinements  
**Severidad**: 🔴 Critical  
**Módulo**: Sessions  
**Tiempo de resolución**: ~45 min

### Síntoma

Error en runtime al acceder a `/dashboard/sessions`:

```
TypeError: undefined is not an object (evaluating 'session.subject.name')
```

Las sesiones se creaban correctamente en la DB pero al mostrarlas en el frontend, crasheaba al intentar acceder a `session.subject.name` o `session.topic.name`.

### Hipótesis Generadas

1. **H1**: El query no trae las relaciones (topic, subject) correctamente
2. **H2**: Las relaciones existen pero están mal mapeadas en los tipos TypeScript
3. **H3**: El componente accede a propiedades incorrectas
4. **H4**: El usuario no está autenticado y el query falla silenciosamente

### Instrumentación

Logs agregados en:
- `sessions.ts:9` - Verificar user ID y parámetros
- `sessions.ts:42` - Verificar resultado del query Supabase
- `session-card.tsx:42` - Verificar datos recibidos en render
- `page.tsx:6` - Verificar estado de autenticación

### Evidencia de Logs

```json
{"location":"session-card.tsx:42","data":{"sessionId":"6f509627...","hasTopic":false,"hasSubject":false}}
```

**Análisis**:
- **H1**: ✅ CONFIRMADA - `hasTopic: false, hasSubject: false` prueba que las relaciones no se traen
- **H2**: ❌ RECHAZADA - No es problema de tipado
- **H3**: ❌ RECHAZADA - El componente accede correctamente
- **H4**: ❌ RECHAZADA - El userId existe en los logs

### Causa Raíz

La sintaxis del `.select()` en Supabase estaba incorrecta. Al listar cada campo individualmente:

```typescript
.select(`
  id,
  topic_id,
  subject_id,
  topic:topics(id, name),
  subject:subjects(id, name)
`)
```

Supabase no ejecuta los JOINs correctamente cuando se mezclan campos explícitos con relaciones.

### Solución

```typescript
// ❌ ANTES (no funciona)
.select(`
  id,
  topic_id,
  subject_id,
  scheduled_at,
  duration,
  priority,
  status,
  number,
  attempts,
  topic:topics(id, name),
  subject:subjects(id, name),
  exam:exams(id, type, date)
`)

// ✅ DESPUÉS (funciona)
.select(`
  *,
  topic:topics(id, name),
  subject:subjects(id, name),
  exam:exams(id, type, date)
`)
```

Usar `*` para los campos de la tabla base + relaciones explícitas.

- [`src/lib/actions/sessions.ts`](../../../src/lib/actions/sessions.ts) - 3 queries corregidos:
  - `getUpcomingSessions()`
  - `getTodaySessions()`
  - `getSessionsByDateRange()`
- [`src/app/(dashboard)/dashboard/sessions/page.tsx`](../../../src/app/(dashboard)/dashboard/sessions/page.tsx) - Agregado null check

### Testing

- ✅ Verificado con logs post-fix (topic y subject presentes)
- ✅ UI funciona correctamente
- ✅ Tests unitarios: 60/60 passing

### Impacto

- **Usuarios afectados**: Todos (feature completamente bloqueada)
- **Scope**: Vista de sesiones + Dashboard
- **Downtime**: ~15 minutos
- **Regresiones**: Ninguna

### Lecciones Aprendidas

1. **Documentación de Supabase**: No es clara sobre sintaxis de JOINs con campos explícitos
2. **Debug sistemático**: Los logs identificaron el problema en segundos (hasTopic: false)
3. **Sintaxis robusta**: Usar `*` es más seguro que listar campos manualmente
4. **Testing**: Los tests unitarios no detectaron esto (solo probaban la lógica, no los queries reales)

**Para evitar en el futuro**:

- Agregar tests de integración que verifiquen queries reales
- Validar con TypeScript que las relaciones existan antes de acceder

## 4.1.2 - Query topics sin user_id

**Fecha**: 28 Enero 2026  
**Sprint**: 4.0 - UI Refinements  
**Severidad**: 🔴 Critical  
**Módulo**: Sessions (Generator)  
**Tiempo de resolución**: ~30 min

### Síntoma

Error al crear un topic:

```
Warning: Could not generate sessions: column topics.user_id does not exist
```

Las sesiones no se generaban automáticamente al crear un tema.

### Causa Raíz

El query en `generateSessions()` intentaba filtrar por `topics.user_id`:

```typescript
.eq('user_id', user.id) // ❌ topics no tiene user_id
```

La tabla `topics` no tiene columna `user_id` directamente. Solo `subjects` la tiene.

### Solución

Validar ownership a través de la relación con `subjects`:

```typescript
// ✅ CORRECTO
.select(`..., subject:subjects!inner(user_id)`)
.eq('subject.user_id', user.id)
```

### Archivos Modificados

- [`src/lib/actions/sessions.ts`](../../../src/lib/actions/sessions.ts) - `generateSessions()` function

### Impacto

- **Usuarios afectados**: Todos (sesiones no se generaban)
- **Scope**: Creación de topics
- **Downtime**: ~30 minutos

### Lecciones Aprendidas

1. Revisar el schema antes de escribir queries
2. No asumir que todas las tablas tienen `user_id`
3. Usar relaciones para validar ownership

## 4.1.3 - Sessions Date Range

**Fecha**: 28 Enero 2026  
**Sprint**: 4.0 - UI Refinements  
**Severidad**: 🟡 Medium  
**Módulo**: Sessions (Display)  
**Tiempo de resolución**: ~10 min

### Síntoma

Las sesiones se creaban correctamente en la DB pero no se mostraban en el frontend.

SQL mostraba inserts exitosos:
```sql
INSERT INTO sessions (...) VALUES 
('...', '2026-02-13 12:00:00+00', ...),
('...', '2026-02-20 12:00:00+00', ...)
```

Pero la UI estaba vacía.

### Causa Raíz

Los queries traían solo sesiones de los próximos **7 días**, pero las sesiones generadas estaban a 15-29 días (fechas en febrero).

```typescript
const endDate = new Date(today);
endDate.setDate(endDate.getDate() + 7); // Solo 7 días
```

### Solución

Ampliar el rango a 30 días:

```typescript
// Dashboard
const thirtyDaysLater = new Date(today);
thirtyDaysLater.setDate(thirtyDaysLater.getDate() + 30);

// Sessions page
const sessionsData = await getUpcomingSessions(30); // en vez de 7
```

### Archivos Modificados

- [`src/lib/actions/dashboard.ts`](../../../src/lib/actions/dashboard.ts) - Query ampliado a 30 días
- [`src/app/(dashboard)/dashboard/sessions/sessions-client.tsx`](../../../src/app/(dashboard)/dashboard/sessions/sessions-client.tsx) - Parámetro `days` aumentado
- UI labels actualizados

### Impacto

- **Usuarios afectados**: Todos
- **Scope**: Visualización de sesiones
- **UX impact**: Mejora significativa (ahora se ven todas las sesiones relevantes)

### Lecciones Aprendidas

1. El rango de fechas debe coincidir con los intervalos de generación
2. Para sesiones Anki ([1,3,7,14]), el máximo es 14 días, pero con finales puede extenderse más
3. 30 días es un buen balance entre "próximas" y "todas"

**Sprint**: 4.0 - UI Refinements

## 4.2.2 - Input onChange Type Error

**Fecha**: 28 Enero 2026  
**Sprint**: 4.0 - UI Refinements  
**Severidad**: 🟠 High  
**Módulo**: Profile  
**Tiempo de resolución**: ~15 min

### Síntoma

Build error:

```
Type error: Property 'value' does not exist on type 'EventTarget & HTMLInputElement'.
```

Múltiples inputs en el formulario de perfil tenían este error.

### Causa Raíz

Configuración estricta de TypeScript no infiere correctamente el tipo del evento:

```typescript
onChange={(e) => setName(e.target.value)} // ❌ e.target puede no ser HTMLInputElement
```

### Solución

Usar `e.target.value` directamente (TypeScript lo infiere correctamente con `<input>`):

```typescript
// ✅ Simple y funciona
onChange={(e) => setName(e.target.value)}
```

No requiere tipado explícito cuando el elemento es `<input>`.

### Archivos Modificados

- [`src/app/(dashboard)/dashboard/profile/profile-client.tsx`](../../../src/app/(dashboard)/dashboard/profile/profile-client.tsx) - 3 inputs corregidos

### Lecciones Aprendidas

1. TypeScript infiere el tipo correctamente para elementos HTML nativos
2. No sobre-complicar con tipado explícito innecesario
3. El error es más común en proyectos con strict mode

## 4.3.1 - Countdown → Free Study Logic

**Fecha**: 28 Enero 2026  
**Sprint**: 4.0 - UI Refinements  
**Severidad**: 🟠 High  
**Módulo**: Session Generator  
**Tiempo de resolución**: ~1 hora

### Síntoma

Las sesiones para finales se generaban muy lejos en el tiempo:

```
Final: 15 Marzo
├─ R1: 1 Marzo (-14 días) ← muy lejos!
├─ R2: 8 Marzo  
├─ R3: 12 Marzo
└─ R4: 14 Marzo
```

El usuario quería poder **empezar a estudiar inmediatamente**, no esperar semanas.

### Causa Raíz

El modo "Countdown" distribuía sesiones ANTES del examen, contando hacia atrás. Esto era lógico teóricamente pero poco práctico:

- Si creás un tema hoy para un final en 2 meses, la primera sesión es en 1.5 meses
- Promovía procrastinación
- No permitía arrancar rápido

### Solución

Cambiar de "Countdown" a "Free Study desde HOY":

```typescript
// ❌ ANTES: Countdown (hacia atrás)
function generateCountdownSessions() {
  const examDate = new Date(exam.date);
  const invertedIntervals = [14, 7, 3, 1]; // días ANTES del examen
  
  invertedIntervals.forEach(daysBefore => {
    sessionDate = examDate - daysBefore;
  });
}

// ✅ DESPUÉS: Forward desde hoy
function generateFreeStudySessions() {
  const today = new Date(); // Arrancar HOY
  const intervals = [1, 3, 7, 14]; // días DESDE hoy
  
  intervals.forEach(daysAfter => {
    sessionDate = today + daysAfter;
  });
}
```

**Resultado**:
```
Hoy: 27 Enero (creas el tema)
├─ R1: 28 Enero (+1) ← ¡mañana!
├─ R2: 30 Enero (+3)
├─ R3: 3 Febrero (+7)
└─ R4: 10 Febrero (+14)
```

### Archivos Modificados

- [`src/lib/services/session-generator.ts`](../../../src/lib/services/session-generator.ts) - Lógica completa reescrita
- [`src/__tests__/unit/services/session-generator.test.ts`](../../../src/__tests__/unit/services/session-generator.test.ts) - Tests actualizados
- [`docs/SESSION_GENERATION_LOGIC_V2.md`](../../SESSION_GENERATION_LOGIC_V2.md) - Nueva documentación

### Testing

- ✅ 60/60 unit tests passing
- ✅ Verificación manual: sesiones desde mañana
- ✅ Casos edge: examen cercano/lejano

### Impacto

- **Usuarios afectados**: Usuarios con finales
- **Scope**: Generación de sesiones para FREE_STUDY
- **UX impact**: 🚀 Mejora dramática - poder arrancar inmediatamente

### Lecciones Aprendidas

1. **UX sobre teoría**: Aunque "countdown" era lógico, no era práctico
2. **Feedback del usuario**: El usuario identificó el problema real
3. **Simplicidad**: Mismos intervalos para ambos modos (más fácil de entender)
4. **Anti-procrastinación**: Primera sesión mañana ayuda a arrancar rápido

**Design principle identificado**: Para apps de productividad, priorizar **acción inmediata** sobre **planificación perfecta**.

## 4.2.3 - Auto Free Study Mode

**Fecha**: 28 Enero 2026  
**Sprint**: 4.0 - UI Refinements  
**Severidad**: 🟡 Medium (Feature, no bug)  
**Módulo**: Topics  
**Tiempo de desarrollo**: ~1.5 horas

### Descripción

Feature nueva: Cuando una materia tiene **solo finales** (sin parciales vigentes), simplificar el formulario de creación de temas automáticamente.

### Implementación

Función de detección:

```typescript
function isAutoFreeStudyMode(exams) {
  const finals = exams.filter(e => e.type.startsWith('FINAL_'));
  const parciales = exams.filter(e => 
    e.type.startsWith('PARCIAL_') || 
    e.type.startsWith('RECUPERATORIO_')
  );
  
  if (finals.length === 0) return false;
  if (parciales.length === 0) return true;
  
  // Si todos los parciales ya pasaron
  return parciales.every(p => new Date(p.date) < new Date());
}
```

**Comportamiento**:
- Oculta campos: Fuente, Fecha de Clase, Examen
- Asigna automáticamente: `source = FREE_STUDY`, `source_date = hoy`, `exam_id = Final`
- Muestra mensaje informativo azul
