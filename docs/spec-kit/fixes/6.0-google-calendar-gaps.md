## Causa Raíz

No hay una "causa raíz" en el sentido tradicional de un bug. El análisis revela que el Sprint 4 se completó exitosamente con el **scope definido inicialmente**: implementar la funcionalidad core de Google Calendar.

**Lo que se implementó (Scope original)**:
- Backend completo para export, import, conflict detection y sync
- UI básica para trigger las acciones (botones, settings)
- Event-driven architecture para mantener calendarios sincronizados

**Lo que quedó fuera del scope (Refinamientos de UX)**:
- Previews visuales antes de importar disponibilidad
- Indicadores visuales de sesiones ajustadas por conflicto
- Feedback toast al completar/reagendar sesiones

**Conclusión**: No es un problema técnico, sino **deuda de UX** acumulada intencionalmente para entregar funcionalidad core primero.

**Modificar Server Action**:
```typescript
// src/lib/actions/availability.ts

export async function importAvailabilityFromGoogleCalendar(
  mode: 'preview' | 'confirm' = 'preview'
) {
  const slots = await availabilityImporter.detectAvailableSlots(...);
  
  if (mode === 'preview') {
    // NO guardar en DB, solo retornar preview
    return { success: true, preview: slots };
  }
  
  // mode === 'confirm' → guardar en DB
  await supabase.from('availability_slots').insert(slots);
  return { success: true, saved: slots.length };
}
```

## Lecciones Aprendidas

1. **Priorización backend-first es válida**: Implementar funcionalidad core primero permitió validar la arquitectura antes de invertir en UX
2. **Documentar gaps explícitamente ayuda**: Este análisis facilita la planificación del próximo sprint
3. **Indicadores visuales son críticos**: Un feature funcional pero invisible es percibido como "no implementado" por usuarios
4. **Preview antes de acciones destructivas**: Importar disponibilidad reemplaza datos existentes, el preview previene errores

**Para futuros sprints**:
- Reservar 20-30% del tiempo del sprint para "polish" de UX
- Definir desde el inicio qué es "MVP del feature" vs "feature completo"
- Agregar migrations de tracking desde el principio (ej: `adjusted_for_conflict`)
- Crear componentes de preview/confirmación para acciones que modifican datos masivamente
