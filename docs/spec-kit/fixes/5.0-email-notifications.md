# 5.0 - Email Notifications Troubleshooting

**Fecha**: 18 Mar 2026  
**Sprint**: 4.0 - Google Calendar Integration  
**Severidad**: 🟡 Medium  
**Módulo**: Notifications  
**Tiempo de resolución**: ~120 min (instrumentación + debugging)

---

## Síntoma

Las notificaciones por email de "sesiones creadas" funcionan correctamente en producción, pero las de "reagendar" y "sesiones vencidas" no se envían.

**Comportamiento observado**:
- ✅ Email enviado al crear topics con sesiones → FUNCIONA
- ❌ Email al reagendar una sesión manualmente → NO FUNCIONA
- ❌ Email para sesiones vencidas (overdue) → NO FUNCIONA

---

## Hipótesis Generadas

1. **H1**: `Promise.allSettled` en NotificationService está tragando errores silenciosamente
2. **H2**: `user_settings.email_notifications` está en `false` para las notificaciones problemáticas
3. **H3**: El `user.id` no se está pasando correctamente en los Server Actions de reschedule/overdue
4. **H4**: `getUserEmail()` devuelve `null` porque el trigger `on_auth_user_created` no sincronizó el email
5. **H5**: El cron job para `processOverdueSessions()` no está configurado en Vercel

## Causa Raíz

### Problema 1: Email no sincronizado (H4 Confirmada)

El trigger `on_auth_user_created` en Supabase no siempre se ejecuta correctamente durante el signup con Google OAuth. Esto causa que la tabla `public.users` no tenga el email del usuario, y `getUserEmail()` devuelve `null`.

**Flujo esperado**:
```
Google OAuth → auth.users creado → TRIGGER → public.users creado con email
```

**Flujo real (bug)**:
```
Google OAuth → auth.users creado → TRIGGER falla/no ejecuta → public.users sin email
```

### Problema 2: Cron job no configurado (H5 Confirmada)

`processOverdueSessions()` es una función que debe ejecutarse periódicamente, pero no hay un cron job configurado en Vercel. La función existe en el código pero nunca se invoca automáticamente.

---

## Solución

### Solución 1: Sincronización manual de emails

**Código Antes**:
```typescript
// ❌ Confiar 100% en el trigger automático
// No había fallback si el trigger falla
```

**Código Después**:
```sql
-- ✅ Script de sincronización manual en Supabase Production
INSERT INTO public.users (id, email, name)
SELECT 
  id, 
  email, 
  raw_user_meta_data->>'name'
FROM auth.users
WHERE id = '<user-id-afectado>'
ON CONFLICT (id) DO UPDATE 
SET email = EXCLUDED.email;
```

### Solución 2: Configurar Vercel Cron Job

**Archivo nuevo**: `vercel.json` (raíz del proyecto)
```json
{
  "crons": [
    {
      "path": "/api/cron/process-overdue-sessions",
      "schedule": "0 0,12 * * *"
    }
  ]
}
```

**Archivo nuevo**: `src/app/api/cron/process-overdue-sessions/route.ts`
```typescript
import { NextResponse } from 'next/server';
import { processOverdueSessions } from '@/lib/actions/sessions';

export async function GET(request: Request) {
  const authHeader = request.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  const result = await processOverdueSessions();
  return NextResponse.json(result);
}
```

**Variable de entorno**: `CRON_SECRET=<random-token>` en Vercel

### Explicación

**Solución 1**: La sincronización manual asegura que todos los usuarios tengan su email en `public.users`, incluso si el trigger automático falló durante el signup.

**Solución 2**: El cron job de Vercel ejecuta automáticamente `processOverdueSessions()` dos veces al día (00:00 y 12:00), enviando notificaciones para sesiones vencidas.

## Lecciones Aprendidas

1. **Logging es crítico**: Sin los logs agregados, el problema habría sido imposible de diagnosticar en producción
2. **No confiar 100% en triggers automáticos**: Siempre tener un fallback o script de sincronización manual
3. **Documentar funciones que requieren cron**: `processOverdueSessions()` existía en el código pero no estaba claro que requería un cron job externo
4. **Promise.allSettled oculta errores**: Aunque es útil para no detener otros canales, dificulta el debugging sin logging explícito

**Para evitar en el futuro**:
- Implementar health check que verifique sincronización de emails durante el signup
- Documentar claramente en el código cuando una función debe ejecutarse vía cron
- Agregar tests E2E que verifiquen el envío de emails en todos los flujos
- Configurar alertas en Vercel si `RESEND_API_KEY` falta o el cron job falla
